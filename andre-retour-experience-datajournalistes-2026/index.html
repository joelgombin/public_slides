Les données électorales françaises

La richesse

  • 53 élections de 1999 à 2024
  • ~70 000 bureaux de vote par élection
  • Données publiques, Ministère de l'Intérieur
  • Granularité géographique unique en Europe
  • Croisements possibles : socio-démo, géographie, historique

La complexité

  • Formats de fichiers changeants d'une élection à l'autre (merci data.gouv pour l'homogénéisation !)
  • Identifiants de BDV instables (changent entre scrutins)
  • Nuances politiques : ~90 codes à interpréter
  • pas d'API officielle, téléchargement manuel

Deux minutes pour montrer pourquoi ces données sont à la fois précieuses et difficiles à exploiter. Les BDV sont particulièrement intéressants car ils permettent des analyses infra-communales qu'on ne peut pas faire avec des données agrégées. Vue d'ensemble architecturale. Public technique donc on peut aller vite. L'important : trois couches clairement séparées. Le MCP (Model Context Protocol) est le point original — c'est le mécanisme qui permet au LLM d'appeler des outils de façon structurée sans permission manuelle. Les "skills" sont des fichiers markdown qui injectent du contexte métier (méthodologie électorale) dans le contexte du LLM. Montrer que c'est du code Python standard sous le capot — rien de magique.

Défi 1 — Qualité des données

Problème : les données électorales publiques sont riches mais imparfaites

  • Doublons dans les Européennes (plusieurs lignes par candidat)
  • Identifiants de bureaux de vote instables entre scrutins
  • Nuances politiques parfois absentes ou incohérentes

Solution :

# Règle de validation systématique dans le code
assert 20_000_000 < total_voix_national < 50_000_000, \
    f"Résultat suspect : {total_voix_national}"

Valider les magnitudes avant chaque réponse : le total des votes nationaux doit toujours être entre 20M et 50M. Si ce n'est pas le cas → bug de JOIN ou doublon.

Joël Gombin

Exemple concret et immédiatement compréhensible pour des journalistes data. Expliquer que cette règle a sauvé plusieurs analyses. La validation des magnitudes est une technique simple mais très efficace pour attraper les bugs silencieux dans les pipelines de données. Mentionner que c'est inspiré des pratiques data engineering (data quality checks). L'idée : ne pas laisser l'agent répondre sur des données potentiellement erronées.

Défi 2 — Hallucinations et garde-fous

Problème : un LLM peut "interpréter librement" des données électorales

Exemple imaginaire mais plausible :

"Le RN a progressé dans ce bureau car c'est une zone ouvrière en désindustrialisation"

Garde-fous d'André :

  • les agrégations par famille politique sont précodées
  • des principes d'analyse sont posés (travailler de préférence aux inscrits, comparer entre scrutins identiques...)
  • l'agent "voit" les cartes
  • mais on ne peut pas garantir à 100% l'absence d'interprétations hasardeuses

Joël Gombin

C'est le garde-fou dont je suis le plus fier. Les nuances politiques sont des données officielles précises — les remapper vers des familles politiques est une décision éditoriale, pas un calcul. André refuse de le faire automatiquement car c'est une interprétation politique, pas factuelle. Expliquer la distinction : calculer (que fait André) vs interpréter (que fait le journaliste). C'est aussi un garde-fou contre les biais potentiels du LLM sur les questions politiques.

Défi 3 — Du CLI perso au service web

Ce que j'attendais : ajouter une API sur un outil CLI existant, ~2h

Ce que j'ai trouvé :

  • Sessions multi-tour : state management non-trivial
  • SSE streaming : envoyer les tokens au fur et à mesure (UX critique)
  • Thread-safety : requêtes concurrentes sur des sessions différentes
  • Docker multi-stage pour un build reproductible
  • Déploiement continu : push GitHub → CapRover automatique

Leçon : le passage de "ça marche sur ma machine" à "ça marche pour d'autres" sous-estime toujours les problèmes d'infrastructure.

Joël Gombin

Anecdote honnête sur la sous-estimation. Le streaming SSE est crucial pour l'UX des agents — sans ça, l'utilisateur attend 30 secondes devant un écran vide. Expliquer brièvement pourquoi les sessions sont complexes : l'agent garde un contexte de conversation, il faut un lock asyncio pour éviter les race conditions. L'audience technique va apprécier ces détails concrets.

Ce que ça change pour le datajournalisme

Pour le praticien

  • Accès à des données complexes sans friction technique
  • Les BDV deviennent accessibles sans Python/SQL
  • Plus de temps pour l'interprétation, moins pour la manipulation
  • Mémoire méthodologique intégrée (skills)

Les limites à ne pas oublier

  • Coût : chaque analyse = tokens Anthropic (~0,05-0,50€)
  • Latence : 10-30s par analyse complexe
  • Garde-fous nécessaires : l'agent fait confiance aux données
  • Pas de raisonnement causal : corrélations, pas explications
  • Opacité : difficile à auditer pour un tiers

Joël Gombin

Équilibrer honnêtement. Les journalistes techniques vont apprécier la liste des limites — c'est ce qui manque souvent dans les présentations sur l'IA. Insister sur "pas de raisonnement causal" : André peut dire "dans ce bureau, le RN a 12 points de plus qu'en 2022" mais pas "parce que". L'interprétation causale reste un travail humain et éditorial.

Ce que j'aurais fait différemment

Architecture : - Commencer par l'API (pas refactoriser après coup) - Tester les pipelines de données AVANT d'écrire l'agent

Données : - Documenter les conventions de nommage dès le départ - Mettre les validations de magnitudes dès le premier jour

IA : - Définir les garde-fous métier AVANT de coder les features - Écrire les instructions de l'agent ("CLAUDE.md") avant le code

Un agent est aussi bon que ses instructions et ses données — pas plus.
