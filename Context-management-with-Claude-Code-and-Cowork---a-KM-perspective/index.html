The big idea

For decades, knowledge management has struggled with one problem: getting people to write down what they know.

AI context files solve this — not because people suddenly want to document, but because the AI doesn't work well without it.

Writing context for AI is writing organizational knowledge. The incentive structure has finally aligned.

"Think of CLAUDE.md as a README, but instead of being written for human developers joining your team, it's written for your AI pair programmer." — Anthropic

Joël Gombin

From prompt engineering to context engineering

Prompt engineering = crafting the right question each time.

Context engineering = building an environment so the AI already understands your standards, terminology, and workflows before you ask anything.

The shift: from individual skill (writing good prompts) to organizational infrastructure (structured knowledge that persists).

Four strategies for managing AI context

Capitalize

Write down what the team knows — conventions, glossaries, procedures — so the AI can reuse it

Target

Give the AI only the information relevant to the current task, not everything at once

Summarize

Condense long conversations and documents so the AI stays focused on what matters

Compartmentalize

Keep different topics in separate spaces so they don't interfere with each other

Context files: your team's knowledge, always loaded

Plain text files (called CLAUDE.md) that the AI reads automatically at the start of every session. Think of them as a style guide the AI actually follows.

Different folders can have different rules. The AI applies the right ones depending on where it's working.

For Sarah: her new writers get the benefit of the team's accumulated knowledge from day one — because the AI already knows the rules.

What belongs in a context file — for a docs team

Include

  • Your style guide essentials (tone, voice, formatting)
  • Terminology: "We say X, never Y"
  • Document structure rules (heading hierarchy, section order)
  • Review workflow: who approves what
  • Common commands to build/preview docs

Avoid

  • Anything a spell-checker can enforce
  • Instructions longer than ~500 lines (the AI loses focus)
  • Generic advice ("write clearly") — the AI already knows
  • Rapidly changing info — use live connectors instead

Key insight: the act of writing these files forces teams to make tacit knowledge explicit — the exact challenge KM has faced for decades.

Memory & Skills: learning and procedures

Memory — what the AI learns

The AI notices patterns and saves them: your naming conventions, your preferred phrasing, past decisions.

This builds up over time — like onboarding a new team member who actually remembers everything.

For Sarah: "We decided in March to use 'user' not 'customer' in all docs" — the AI remembers, even months later.

Skills — step-by-step procedures

Reusable procedure guides the AI loads on demand. Each one describes how to do a specific task.

Examples for a docs team: - "How to write a changelog entry" - "How to review a document for consistency" - "How to generate API reference docs"

For Sarah: instead of training each writer, she writes the procedure once. The AI follows it every time.

Connectors (MCP): linking to your existing tools

Creation & Structure

Creation

Without context: each writer prompts the AI from scratch. Output varies wildly.

With context: the AI loads your style guide, templates, and glossary automatically. First drafts already follow your standards.

Sarah's setup: - Context file with style guide rules - One Skill per document type (user guide, API ref, changelog) - Connector to GitHub for live code examples

Structure

Without context: the AI makes up its own document structure every time.

With context: heading hierarchy, section order, and depth limits are encoded in your rules. Different doc types follow different templates.

What to watch out for: - Don't encode rules that change monthly — use connectors for dynamic content - Start with 5 rules, not 50 — you can always add more - Test with a real document before rolling out

Joël Gombin

Search, Consistency & Readability

Search

Connect your knowledge base (Notion, Confluence) via connectors.

Create a project space with your key references uploaded.

Team members ask: "Where's our doc on X?" or "What did we decide about Y?"

Result: unified search across tools + institutional memory

Consistency

Maintain a terminology glossary and reference it in your context file: "Always use terms from our glossary."

Create a "review for consistency" procedure as a Skill.

The AI learns your team's specific patterns over time via Memory.

Result: consistency improves as the system learns

Readability

Define audience personas in your context file: "Our users are non-technical. Use short sentences."

Create a readability review procedure.

Set automatic checks after each edit.

Result: measurable readability improvements

Choosing your setup

There's no one-size-fits-all. Here are three architectures matched to team profiles:

Small docs team, no developers

Claude.ai Projects + Cowork — Upload your style guide, create a shared project, start writing. No setup required beyond a subscription.

Mixed team (writers + developers)

Cowork + Claude Code + Connectors — Writers use Cowork for content, developers use Claude Code for automation. Connect to your tools via MCP.

Large organization, publishing pipeline

Claude Code + automated checks + custom connectors — Full automation: context files version-controlled in Git, automatic quality checks in your CI/CD pipeline, custom connectors to internal systems.

Getting started: your first month

Week 1-2: Foundation

  • Write a context file with your 5 most important style rules and your terminology glossary
  • Create a Claude.ai Project and upload your style guide
  • Test: ask the AI to draft a document you'd normally write yourself

Week 3-4: Procedures

  • Write one Skill for your most common document type
  • Connect one external tool (Notion, Confluence, or GitHub)
  • Have a second team member test the setup independently

Month 2: Expand

  • Add context files for different doc types (tutorials vs. API ref vs. guides)
  • Create a "doc review" Skill
  • Start using Memory — let the AI learn your patterns

Month 3+: Automate

  • Version-control your context files in Git
  • Add automatic quality checks
  • Audit and refine your context files quarterly — treat them like living documentation

Key takeaways

  • Context is strategy, not a feature. The team that structures its AI context best gets qualitatively better results — not just faster output, but output that follows standards from the start.

  • Context files are a new kind of knowledge asset. They deserve the same governance as your style guide: versioned, reviewed, maintained.

  • The KM opportunity is real. Writing AI context forces the externalization of tacit knowledge. For the first time, there's a direct incentive to document what your team knows.

  • Start small, start now. Five style rules in a text file + a Claude.ai Project = your minimum viable setup. Everything else builds from there.

What you can do Monday morning

Three actions, no technical skills required:

1. Open Claude.ai and create a Project for your docs team Upload your style guide and glossary as Project Knowledge.

2. Write 10 lines of context Your 5 non-negotiable style rules + 5 terminology choices ("We say X, never Y").

3. Test it with a real task Ask the AI to draft a document type you produce regularly. Compare the output with and without context.

That's it. The rest is iteration.
